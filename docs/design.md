# 设计理念

> 这里记录 wiki_scan 背后的核心设计决策，帮助贡献者理解「为什么这样设计」。

---

## 核心洞察：AI 读代码和人读代码需要的不同

**人读代码需要**：叙事、上下文、因果（"为什么这样做"）
**AI 用代码干活需要**：精准片段、字段名、接口契约、约束边界

这是 wiki_scan 双轨设计的根本原因：把两类知识严格分离，互不干扰。

---

## 为什么分 L0-L3 四层？

这是对"AI Token 预算"的系统性回答。

| 层 | 大小目标 | 读取时机 |
|---|---|---|
| L0（技术画像）| < 2KB | 每次任务必读 |
| L1（架构） | < 20KB | 新任务/架构决策时 |
| L2（详情） | 按需，5-50KB | 改某个具体模块时 |
| L3（约束/未知）| < 10KB | 审查/边界确认时 |

规则：**AI 每次只读当前任务所需的最小文档子集**。`_index.json` 是实现这一目标的路由机制。

---

## 为什么有 L3_unknown.md？

这是 wiki_scan 最独特的设计，没有其他工具做过：

传统 AI 代码助手的危险模式：AI 读到一段复杂的、没有注释的代码，直接"理解"后修改，结果引入Bug。

`L3_unknown.md` 让 AI 必须承认"这段代码我看不懂它的业务意图"，并且明确声明"在用户确认之前，我不会动它"。

这把"AI 的无知"从隐患变成了显式的防护机制。

---

## 为什么要区分「操作清单」和「业务规则清单」？

代码只能告诉你"系统做了什么"（操作），告诉不了你"系统为什么这样做"（规则）。

例如：
- **操作**：`WHERE audit_status IN (3, 5)` ← 代码可以读出
- **业务规则**：`3 代表已发布，5 代表草稿，草稿岗位也显示因为 HRD 需要编辑` ← 代码读不出，需要人回答

双清单机制让 AI 知道"自己知道什么"和"自己不知道什么"，后者自动转化为 `BUSINESS_QA.md` 的问题，引导用户填写。

---

## 为什么有适配器层？

最初版本（v3）把所有 Java/Spring 的命令硬编码在主 Skill 里。
别拿去用 Python 项目，完全不工作。

适配器是"技术栈插件"：主 Skill 是通用协议，适配器是框架实现。
这样可以在不修改核心逻辑的情况下支持新技术栈。

---

## 已知问题 / 下一步演进

1. **知识腐败问题**：代码变化时，wiki 不会自动更新。需要 `_dependency_map.json` + CI hook。
2. **大型项目知识过大**：L2 文件在 500 文件+项目里会超出 Token 限制。需要 L2.5 方法卡层。
3. **BUSINESS_QA 闭环缺失**：答案填写后没有自动归档机制，需要 L4 知识确认层。
